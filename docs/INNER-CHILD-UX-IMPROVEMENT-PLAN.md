## 1. A. 문항 단축 — 채점 수학과 함께

### 1-1. 현재 문항 수 실측

| 파트 | 문항 | 비고 |
|---|---|---|
| 스크리닝 | 8 | 영역당 2(정서+행동) |
| 공통 C1 | 1 | entitlement_score |
| 드릴다운 | **6~9** | 상위 2영역 조합에 따라: 단절(5)+자율성(4)=9 ~ 타인중심(3)+과잉경계(3)=6 |
| 지킴이 | 3 | 3지선다 |
| SCT | 5 | 자유 입력 |
| **합계** | **23~26** | 인트로 카피의 "22~26"은 드릴다운 최소치를 5로 잡은 설계 문서 기준 — 실제 최소는 6 |

### 1-2. 각 레버의 채점 영향 분석

**레버 1. 스크리닝 8→4 (영역당 1문항)** — ❌ 기각
- `scoreAreas()`의 동점 처리 1단계가 정서 슬롯(S1/S3/S5/S7) 점수인데, 영역당 1문항이면 "정서 슬롯 = 영역 점수 그 자체"가 되어 동점 처리가 무력화된다.
- 영역 점수 범위가 2~12 → 1~6으로 줄어 동점 자체가 급증(4영역이 1~6 안에 몰림).
- 영역 점수는 `selectChildren()`의 대표 아이 동점 처리 ⓐ에도 쓰인다 — 해상도 반토막은 대표 아이 판정까지 흔든다.
- 스크리닝은 전체 적응 로직의 **입력 신호**라 여기서 아끼면 하류 전부가 흔들린다. 8 유지.

**레버 2. 드릴다운을 상위 1영역만 출제** — △ 대안으로만 (아래 1-4)
- 문항 3~5개 절감. 그러나 `secondary_children`이 같은 영역 안에서만 나온다.
- 유료 리포트 섹션 3(두 번째 아이)과 `second_child_relation` 프롬프트는 conditional 조합("무조건적 A를 막으려 조건적 B가 발달")으로 서사를 만드는데, 같은 영역의 도식은 conditional 값이 대부분 동일(단절·자율성=전부 무조건적, 타인중심=전부 조건적)이라 **유료 상품의 서사 다양성이 눈에 띄게 줄어든다.**
- 대표 아이 자체는 유지된다(영역 1위가 이미 최강 신호라는 전제하에) — 다만 2위 영역의 최고점 도식이 1위 영역 문항들을 이기는 케이스(영역 합산은 밀렸지만 단일 도식이 강한 경우)를 놓친다.

**레버 3. 드릴다운 영역당 3개로 캡** — ❌ 기각
- 도식 1개 = 문항 1개라서, 캡 = 특정 도식(=유형카드=아이)을 **측정 불가능**하게 만드는 것. 16종 유형카드가 상품 자산인데 일부 아이가 영원히 안 나오는 구조가 된다.

**레버 4. SCT 5→3** — ✅ 권장 (핵심 컷)
- 리포트 프롬프트가 실제로 인용하는 슬롯을 전수 확인한 결과:

| SCT 슬롯 | 무료(§5-2 / `free-report-prompt.ts`) | 유료(§6-4 / `paid-report-prompt.ts`) | 판정 |
|---|---|---|---|
| `childhood_self` (SCT1) | 인용 우선순위 **1위** | `core_need_bridge` 인용 (family_rule과 택1) | **유지** |
| `inner_voice` (SCT2) | 인용 우선순위 3위(폴백의 폴백) | 직접 인용 없음 | **컷** |
| `family_rule` (SCT3) | 없음 | `core_need_bridge`에서 childhood_self의 **대체재**로만 | **컷** |
| `regression_trigger` (SCT4) | 인용 우선순위 2위 | `loop_narrative` 촉발 슬롯 **원문 인용 필수** | **유지** |
| `escape_behavior` (SCT5) | 없음 | `guardian_anatomy` **필수** + loop 복선 + `closing` 구체화 — 3곳 | **유지** |

- 즉 SCT2·SCT3은 프롬프트상 "있으면 좋은" 슬롯이고, SCT1·4·5가 하중을 전부 진다.
- 트레이드오프: ⓐ 무료 인용 체인이 2단(childhood_self → regression_trigger)으로 줄어 둘 다 빈약할 때 인용 품질 저하 가능 → 프롬프트에 "인용할 만한 응답이 없으면 인용 생략" 분기 추가로 방어. ⓑ `core_need_bridge`의 대체 인용원 상실 → childhood_self 단독으로 변경. ⓒ 위기 필터 스캔 필드 5→3 — `inner_voice`는 위기 문구("죽어버려" 류)가 나올 수 있는 슬롯이라 커버리지가 실제로 준다. **완화책: 남는 SCT 3개 + (선택) 자유 입력이 생기는 모든 필드를 스캔 대상으로 유지하고, 위기 키워드 리스트를 보강.**
- SCT는 **타이핑 파트**라 문항당 체감 비용이 6점 척도의 2~3배다. 2문항 컷의 체감 효과는 숫자 이상.

**레버 5. 지킴이 3→2** — ❌ 기각
- 2문항이면 다수결이 성립하지 않고(2:0 아니면 1:1), 1:1일 때 항상 G2 채택 = 사실상 G2 단독 판정. 판별 안정성 대비 절감 1문항은 수지가 안 맞는다. 게다가 3지선다는 §2의 "포맷 다양성" 자산이라 피로 기여도가 낮다.

**레버 6. SCT 건너뛰기 허용** — ✅ 병행 권장 (문항 수 불변, 이탈 방어)
- SCT는 최대 이탈 지점(타이핑 요구). 문항별 "건너뛰기" 허용하되, 프롬프트에 빈 슬롯 폴백("해당 인용 없이 일반 서술") 명시. 어차피 사용자가 성의 없는 입력을 넣는 경우에 대한 프롬프트 방어가 필요하므로 겸사겸사 해결.

### 1-3. 권장안 (안 A): 21~24문항

| 파트 | 현재 | 변경 | 문항 |
|---|---|---|---|
| 1부 스크리닝 + C1 | 8+1 | 유지 (C1을 1부에 시각적으로 편입) | 9 |
| 2부 드릴다운 | 6~9 | 유지 (적응의 본체 — 여기가 "파고드는 느낌"의 원료) | 6~9 |
| 2부 지킴이 | 3 | 유지 | 3 |
| 3부 SCT | 5 | **3** (SCT2 inner_voice, SCT3 family_rule 컷) + 건너뛰기 허용 | 3 |
| **합계** | 23~26 | | **21~24** |

- 절대 절감은 2문항이지만, 컷 대상이 타이핑 문항이라 체감 절감이 크고, §2의 챕터·인터스티셜·포맷 다양화가 체감 길이를 추가로 줄인다. "길다"는 피드백의 절반은 문제 3(단조로움)의 그림자라는 게 이 플랜의 판단.
- 채점 리스크: **낮음.** `computeScore`의 위기 스캔 필드와 `SctAnswers` 타입만 변경. 영역/대표/지킴이 로직 무변.
- 인트로 카피 "22~26개 문항(약 4분)" → "약 20여 개 문항(약 3분)"으로 갱신. `TOTAL_ESTIMATE`(현재 24) → 22로 조정.

### 1-4. 대안 (안 B): 18~20문항 — 공격적

- 안 A + **드릴다운을 상위 1영역만 출제**(`TOP_AREAS_COUNT` 2→1): 드릴다운 3~5문항.
- 합계: 9 + (3~5) + 3 + 3 = **18~20**.
- 채점 리스크: **중~높음.**
  - `secondary_children`이 동일 영역 한정 → 유료 섹션 3 서사 약화(위 레버 2 분석).
  - 2위 영역의 강한 단일 도식을 놓치는 케이스 발생.
  - `scoring.test.ts`의 `selectChildren` 계열 테스트(상위 2영역 전제 픽스처)가 다수 수정 대상.
- **채택 기준 제안**: 안 A로 파일럿(지인 10~20명, Phase 7) 돌리고, "길다" 피드백이 여전히 우세하면 안 B로 후퇴. 처음부터 안 B로 가면 유료 리포트 품질 저하 원인이 문항 수인지 판별할 수 없게 된다.

## 2. B. 적응형 체감 설계 — 핵심 과제

> 원인: 스크리닝과 드릴다운이 같은 컴포넌트(`ScaleQuestion`), 같은 6점 원형 버튼, 같은 진행바 위에서 이어진다.
> `appendAdaptiveSteps()`가 상위 2영역을 계산해 문항을 갈아끼우는 순간이 **사용자에게 0px도 보이지 않는다.**

### 2-1. 3부 챕터 구조 (진행바 재설계)

플랫 진행바 → **챕터 라벨 + 챕터 내 진행** 조합. 진행바는 3분할 세그먼트로.

| 챕터 | 라벨 | 구성 | 인터랙션 텍스처 |
|---|---|---|---|
| 1부 | **마음의 지형** | 스크리닝 8 + C1 (9문항) | 6점 도트 척도 (현행 스타일) |
| 2부 | **더 가까이** | 드릴다운 6~9 + 지킴이 3 | 6단 슬라이더 + 장면 카드 3지선다 |
| 3부 | **당신의 이야기** | SCT 3 | 문장완성 타이핑 |

- 화면 상단: `1부 · 마음의 지형` 킥커 + `4/9` 모노 카운터. 챕터가 바뀔 때만 세그먼트가 채워진다.
- "몇 문항 남았는지"의 불확실성(적응형이라 총량 가변)을 챕터 단위로 흡수 — 챕터 안에서는 분모가 확정적이므로 진행바가 요동치지 않는다. 현재 `TOTAL_ESTIMATE` 고정 분모 방식의 어색함(드릴다운 6개일 때 바가 100% 전에 끝남)도 함께 해소.

### 2-2. 전환 인터스티셜 — 좁혀 들어가는 순간을 보여주기

**인터스티셜 ① 스크리닝 → 드릴다운 (C1 응답 직후, `appendAdaptiveSteps` 시점)**

여기가 이 플랜의 심장이다. 시스템이 방금 사용자를 "읽었다"는 사실을 화면으로 선언한다.

- **스포일러 리스크 판단**: 영역명 원문("단절·거절")은 HANDOFF 확정사항상 **소비자 노출 금지 이론 용어**라 어차피 못 쓴다. 대신 영역별로 뭉툭한 "신호 문구"를 두면 — 결과의 본체(아이 이름·유형)는 전혀 새지 않으면서 "내 답을 듣고 있었네"라는 감각만 준다. 무료 리포트의 히어로는 아이 이름이므로 영역 결의 문구는 스포일러가 아니라 **예고편**이다. → 영역 맛 문구 채택 권장. (더 보수적으로 가려면 아래 무색 버전 사용.)

영역별 신호 문구(1위 영역 기준, `questions.ts`에 상수로 추가):

| topAreas[0] | 문구 |
|---|---|
| disconnection | "응답 속에서, **사람들과의 사이에 놓인 거리** 쪽에서 신호가 잡혔어요." |
| impaired_autonomy | "응답 속에서, **혼자 서야 하는 순간의 막막함** 쪽에서 신호가 잡혔어요." |
| other_directedness | "응답 속에서, **나보다 상대가 먼저 보이는 마음** 쪽에서 신호가 잡혔어요." |
| overvigilance | "응답 속에서, **놓지 못하는 긴장** 쪽에서 신호가 잡혔어요." |

공통 후행 문장 + CTA:

> "지금부터는 모두에게 묻는 질문이 아니에요. **당신의 응답이 고른 질문**이에요. 이 신호가 어느 아이의 것인지, 조금 더 가까이 들어가 볼게요."
> [계속 →]

무색(스포일러 제로) 대안: "응답이 한쪽으로 또렷하게 기울고 있어요. 남은 질문은 그 방향으로만 골라서 드릴게요."

**인터스티셜 ② 드릴다운 → 지킴이**

> "아이의 신호는 충분히 모였어요. 이제 방향을 바꿔서 — 그 아이가 힘들 때, **당신이 스스로를 지켜온 방식**을 볼게요. 이번엔 점수가 아니라 장면이에요."
> [계속 →]

"이번엔 점수가 아니라 장면이에요"가 포맷 전환(6점→3지선다)을 명시적으로 선언 — 단조로움이 깨졌음을 사용자가 인지하게 만든다.

**인터스티셜 ③ 지킴이 → SCT**: 기존 `SCT_TRANSITION`("거의 다 왔어요. 이제 점수가 아니라 당신의 이야기를 들려주세요.")을 인라인 문구에서 **독립 인터스티셜 화면으로 승격**. 3부 챕터 전환과 겹치므로 별도 카피 신작 불필요.

- 구현: `Step` 유니언에 `{ kind: "interstitial"; copy: ... }` 추가. 응답을 요구하지 않는 탭-스루 화면이라 채점 입력엔 무영향. 문항 카운터에서 제외.

### 2-3. 포맷 다양성 — 데이터 제약을 정직하게

**제약 먼저**: 드릴다운 점수는 도식 간 **직접 비교**되어 대표 아이를 뽑는다(`selectChildren`). 문항마다 응답 형식(양자택일, 빈도 척도 등)이 다르면 점수의 비교 가능성이 깨진다. 스크리닝도 영역 합산·동점 처리가 1~6 해상도를 전제한다. 따라서 **수치 파트의 데이터 포맷은 6점 유지, 다양성은 "보이는 껍질"과 파트 경계에서** 만든다:

| 파트 | 인터랙션 | 데이터 매핑 |
|---|---|---|
| 1부 스크리닝+C1 | 현행 6점 **원형 도트** 유지(크기 그라디언트로 방향성 시각화: 왼쪽 작게→오른쪽 크게) | `ScaleValue` 그대로 |
| 2부 드릴다운 | **6단 디텐트 슬라이더**(또는 6칸 세그먼트 바). 라벨을 "전혀 나와 다르다/완전히 나다" → "이 아이는 나와 멀다 / 나에 가깝다"로 교체해 '가까이 들어간다' 프레임 강화 | 디텐트 위치 = `ScaleValue` 1~6, 채점 무변 |
| 2부 지킴이 | 현행 3지선다를 **장면 카드**로: 프롬프트를 "이런 장면을 떠올려 보세요 —" 도입 + 카드형 선택지 | `GuardianType` 그대로 |
| 3부 SCT | 문장완성 타이핑 + 건너뛰기. `______` 자리를 인라인 입력칸으로(현재는 textarea 별도 — 문장 "완성" 감각 강화) | `SctAnswers` 그대로 |

- 양자택일(this-or-that)은 지킴이처럼 **범주형 판별**에만 적합 — 수치 파트에 도입하면 해상도 손실로 동점이 폭증하므로 이번엔 제외. (추후 스크리닝 개편 시 검토 항목으로만 남김.)
- 파트 내 랜덤 배치(HANDOFF 2-1) 구현 시 드릴다운은 상위 2영역 문항을 **섞어서** 출제(영역 블록이 티 나지 않게). `selectChildren`의 동점 처리 ⓒ `order`는 렌더 순서가 아니라 `DRILLDOWN_ITEMS` 은행 순서를 쓰므로 랜덤 렌더가 채점 결정성을 해치지 않음 — 확인 완료.

### 2-4. 마이크로 개인화 (LLM 없음, 전부 클라이언트)

1. **응답 에코**: 인터스티셜 ①에서 스크리닝 최고점 문항 원문을 인용 — "아까 '{최고점 문항 텍스트}'에 크게 공감하셨죠. 그 언저리부터 볼게요." 사용자 자신의 응답이라 스포일러 아님. 최고점 동점이면 정서 슬롯 우선(채점 규칙과 일치).
2. **드릴다운 중간 격려 마이크로카피**(문항 위 작은 라벨, 2~3문항마다): "점점 또렷해지고 있어요" / "거의 다 좁혀졌어요" — 수렴 중이라는 감각.
3. **지킴이 파트 도입을 1위 영역 톤에 맞춰 미세 조정**은 카피 유지비 대비 효과 낮음 → 제외.

## 3. C. 결과 화면

### 3-1. 즉시: dev 하네스 정리 (`src/app/dev/inner-child/page.tsx`)

- 요약 Row(대표 아이·지킴이·영역 순위 등)는 유지 — 디버깅 가치가 핵심.
- `원본 JSON` `<pre>` 블록을 `<details><summary>원본 JSON (디버그)</summary>` 뒤로 접기. 기본 접힘.
- Row 위에 **리포트 프리뷰 섹션** 추가: 유형카드 고정 필드(one_liner, voice, traits, auto_thoughts, triggers)만으로 §3-2 구조를 목업 렌더. LLM 생성 필드(gap, relation_pattern) 자리는 점선 placeholder("생성 구간 — Phase 4"). 이러면 dev 프리뷰가 "제품의 미리보기"로 읽히고, Phase 4에서 이 목업이 실컴포넌트의 뼈대가 된다.

### 3-2. Phase 4: 무료 리포트 화면 구조 (HANDOFF §5-1 8카테고리)

컴포넌트 트리 제안 (`src/components/minds/inner-child/report/` 신설):

```
InnerChildFreeReport            # 조립 + 면책 문구 하단 고정
├── ChildHeroCard               # ① 유형 판정: 일러스트 + child_name + one_liner + core_belief 한 줄 + 태그 칩. 캡처/공유 슬롯
├── VoiceQuote                  # ② 내면의 목소리: 세리프 인용 블록 (본문 고딕과 서체 분리 — 아이가 '다른 존재'로 들리게)
├── TraitsSection               # ③ 기본 성향: traits 3~4문장 프로즈, 강점 선행
├── ThoughtChips                # ④ 자주 하는 생각: auto_thoughts 3개 말풍선 칩 (문단 나열 금지)
├── GapSection                  # ⑤ 겉과 속: LLM gap — "외부에서는 ~ / 내부적으로는 ~" 대비 레이아웃
├── RelationPatternSection      # ⑥ 관계 패턴: LLM relation_pattern, SCT 인용은 VoiceQuote와 동일한 세리프 인용 블록 재사용
├── StressSignals               # ⑦ 스트레스 신호: triggers 3개 리스트 + 지킴이 표면 반응 1줄
└── LockCta                     # ⑧ 잠금: 질문 헤드라인({symbol_action}) + 티저 목차 4개(fixed-texts.ts 템플릿) + ₩9,900 버튼
```

- 디자인 언어: `/minds` 콰이엇 에디토리얼 토큰(`M`) 그대로. HANDOFF §8 준수 — 텍스트 기본, 컬러는 의미 슬롯만(아이=보라 계열), LLM은 텍스트만 생성하고 HTML은 템플릿 주입.
- 테스트 종료 → 리포트 사이에 **생성 대기 화면**(스켈레톤 + "당신의 응답 21개를 읽고 있어요" 카피) 1장 필요 — flash 호출 지연(2~5초)을 서사로 흡수. 이것도 "분석하고 있다"는 체감의 일부.

## 5. 리스크 메모

- **위기 필터 커버리지 축소**(SCT 5→3): `inner_voice` 슬롯이 위기 문구의 유력 출처였음. 잔존 3슬롯 스캔 유지 + `crisis-words.ts` 키워드 보강으로 완화. 파일럿에서 SCT 원문 샘플 검수.
- **인터스티셜 ①의 예고 강도**: 영역 맛 문구가 결과 기대를 특정 방향으로 프라이밍할 수 있음. 파일럿에서 "결과가 예상대로였다(김샜다)" 반응이 나오면 무색 버전으로 폴백 — 문구만 교체하면 되도록 상수화해 둘 것.
- **체감 길이 vs 실제 길이**: 인터스티셜 3장은 화면 수를 늘린다(문항은 아님). 각 1~2문장 + 단일 CTA로 3초 안에 지나가게 유지. 애니메이션 과투자는 금지(콰이엇 에디토리얼 톤 유지).
